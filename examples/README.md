# Examples

This directory will contain worked examples of Aim Praxis applied to specific projects. Each example documents how the methodology's layers, artifacts, and agents played out in a real build.

## Planned: Ship Audit

The first complete worked example will be Ship Audit, the SMB e-commerce shipping invoice audit SaaS being built by Mark Schilling. The Ship Audit example will document:

- The system aim and how it was articulated
- The BMC and VPC at each major iteration
- The story map evolution across release slices
- The BDD scenarios and the ubiquitous language as they grew
- The ATDD acceptance criteria and gate enforcement
- The TDD discipline and coding agent integration
- The CI/CD pipeline and observability integration
- The compounding artifacts as they accumulated
- The rework register's classification and the PDSA agent's contributions
- The metrics across the three Deming levels

The example is not yet written. It will be drafted progressively as Ship Audit reaches significant milestones, so that each section reflects actual experience rather than projection.

## Future examples

Additional examples will be added as the methodology is applied beyond Ship Audit. Candidates include:

- Tower Grove Pottery's operational systems (Mark's pottery business)
- Consulting engagements where Aim Praxis was applied to a client's product
- Open-source projects where the methodology was applied

Each example will follow a consistent structure to support comparison across applications.

## Example structure

Each example will be its own subdirectory containing:

- `README.md` — overview and current status
- `aim.md` — the system aim for this specific project
- `bmc.md`, `vpc.md` — the canvas documents at major iterations
- `story-map-evolution.md` — how the map developed
- `glossary.md` — the project's ubiquitous language as it grew
- `lessons-learned.md` — patterns captured during the build
- `rework-register-summary.md` — classified failures and remediations
- `metrics.md` — measurements at the three Deming levels

## Why worked examples matter

The methodology's credibility depends on worked examples. A methodology with no examples is theory. A methodology with one example is plausible. A methodology with multiple examples across different contexts is established.

The discipline of writing the examples honestly — including failures, dead ends, and adjustments — is also part of how the methodology evolves. Each example surfaces tensions in the methodology that the abstract documents don't reveal.
